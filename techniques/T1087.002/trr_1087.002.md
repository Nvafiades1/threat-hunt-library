## T1087.002: Active Directory Account Discovery

### Technique Overview
T1087.002 revolves around how attackers discover user and group accounts in Active Directory (AD). They typically rely on standard Windows commands such as `net user` or `net group`, custom scripts, or third-party utilities to enumerate information about who’s in the environment and which accounts hold special privileges. In Azure AD environments, attackers perform similar enumerations using Microsoft Graph APIs or Azure AD PowerShell modules.

### Technical Background
Active Directory is commonly set up in a way that grants authenticated users broad read access to directory information. Even low-privileged accounts can query user and group details, which means an adversary who has established minimal credentials can still map out the network’s identity layout. This includes pinpointing which accounts are associated with elevated roles and discovering how different groups and organizational units are structured. These default permissions serve legitimate administrative needs but also provide attackers with a straightforward method of reconnaissance.

Many organizations now operate in hybrid modes, synchronizing on-premises AD with Azure AD. When this happens, attackers can piece together information from both the local and cloud environments, building a comprehensive view of the organization’s directory infrastructure. On the cloud side, the Azure AD PowerShell module or direct access to Microsoft Graph can produce user listings, group memberships, and role data that mirrors what’s available on-prem.

Protocols and methods used by attackers to perform these queries include LDAP (commonly on ports 389 and 636), SAMR and RPC, and Win32 NetApi32 calls such as `NetUserEnum`. These interfaces exist to facilitate administrative tasks but also enable extensive enumeration. In Azure AD, the Microsoft Graph API is often a go-to choice for scripted or automated queries, making it possible to harvest identity data at scale with only basic cloud permissions.

### Procedure A: Enumerate AD with Built-In Windows Commands

**ID:** TRR0000.WIN.A  
**Tactic:** Discovery

1. **Net.exe Commands**  
   - `net user /domain`  
   - `net group "<GroupName>" /domain`  
   - **Usage Example**:
     ```bash
     net user /domain
     net group "Domain Admins" /domain
     ```
   **Why**: Quick, no extra installs, looks like legitimate admin usage.  

   **APIs Used & When**:  
   - **Win32 NetApi32.dll** (e.g., `NetUserEnum`, `NetGroupEnum`)  
   - Communicates via **RPC** to the domain controller.  
   - **Used** any time an attacker runs `net.exe` in a domain context.

2. **PowerShell Cmdlets (Built-In)**  
   *(Requires the AD module—part of RSAT.)*

   - **Get-ADUser**:
     ```powershell
     Get-ADUser -Filter * -Properties SamAccountName,MemberOf
     ```
   - **Get-ADGroupMember**:
     ```powershell
     Get-ADGroupMember -Identity "Domain Admins"
     ```
   **Why**: Can retrieve advanced attributes with minimal lines of code.

   **APIs Used & When**:
   - **.NET ADSI / LDAP**:  
     - Under the hood, these cmdlets issue LDAP queries.  
   - **Used** whenever a PowerShell AD cmdlet is invoked and the user has domain privileges.

### Procedure B: Enumerate AD with Third-Party Tools & Scripts

**ID:** TRR0000.WIN.B  
**Tactic:** Discovery

1. **BloodHound / SharpHound**  
   - **BloodHound**: A GUI for graph-based AD relationship mapping.  
   - **SharpHound**: Data collector enumerating users, groups, and sessions.
   - **Example**:
     ```powershell
     .\SharpHound.exe --CollectionMethods All --JSON
     # Import JSON into BloodHound
     ```

   **APIs Used & When**:
   - **LDAP**, **SAMR**, **DCERPC** calls to gather domain details.  
   - **Used** by attackers for a graph-based approach to identify shortest paths to high-priv accounts.

2. **PowerView (PowerSploit)**  
   - PowerShell module for advanced AD recon.
   - **Example**:
     ```powershell
     Import-Module .\PowerView.ps1
     Get-NetUser -Domain edwardjones.local
     ```
   **APIs Used & When**:
   - Primarily **LDAP** queries, some **SAMR** calls.  
   - **Used** if attackers have a PowerShell environment with these scripts loaded.

3. **CrackMapExec**  
   - Multipurpose pentesting tool that enumerates domain objects, tests creds, etc.
   - **Example**:
     ```bash
     crackmapexec smb <DC_IP> -d edwardjones.local -u <User> -p <Pass> --users
     ```
   **APIs Used & When**:
   - Over **SMB** + **SAMR** or **DCERPC**.  
   - **Used** when an attacker wants a single interface for verifying credentials and enumerating domain info.

4. **Soaphound**  
   - A lesser-known tool for enumerating AD objects, possibly via SOAP-based endpoints.
   - **Example**:
     ```powershell
     .\Soaphound.exe --enum-users --domain edwardjones.local
     ```
   **APIs Used & When**:
   - Possibly SOAP or custom RPC calls to AD or domain controllers.  
   - **Used** when standard LDAP or SAMR queries are blocked, but SOAP endpoints remain open.

### Procedure C: Enumerate AD Using Misc. Utilities (AdFind, DsQuery, etc.)

**ID:** TRR0000.WIN.C  
**Tactic:** Discovery

1. **AdFind**  
   - Command-line tool from joeware.net  
   - **Example**:
     ```bash
     AdFind.exe -sc trustdmp -f name=*
     AdFind.exe -b "DC=edwardjones,DC=local" -f "(&(objectCategory=User))"
     ```
   **APIs Used & When**:
   - **LDAP** over 389 or 636.  
   - **Used** if attacker wants advanced or bulk attribute queries.

2. **DsQuery / DsGet**  
   - Built into older Windows Server RSAT.
   - **Example**:
     ```bash
     dsquery user -name *
     dsget group "CN=Domain Admins,CN=Users,DC=edwardjones,DC=local" -members
     ```
   **APIs Used & When**:
   - Under the hood, these call **LDAP** or **ADSI**.  
   - **Used** in older server environments or legacy scripts.

3. **Enum4linux (Linux-based)**  
   - Gathers Windows host info (users, groups) via SMB.
   - **Example**:
     ```bash
     enum4linux -U -S yourdc.edwardjones.local
     ```
   **APIs Used & When**:
   - **SMB** + **DCERPC** / **SAMR**.  
   - **Used** if attacker is pivoting from a Linux environment.

4. **Impacket Scripts (samrdump, lookupsid, etc.)**
   - Python-based queries over SMB/RPC.
   - **Example**:
     ```bash
     python samrdump.py edwardjones.local/username:password@yourdc
     ```
   **APIs Used & When**:
   - **SAMR** (MS-SAMR) calls.  
   - **Used** in Python-based pentest frameworks or custom scripts.

### Procedure D: Enumerate Azure AD Accounts / Groups (Soaphound mention + Detailed APIs)

**ID:** TRR0000.AZURE.D  
**Tactic:** Discovery

For **cloud** or **hybrid** setups leveraging **Azure AD**:

**Azure AD PowerShell**  
   ```powershell
   Connect-AzureAD
   Get-AzureADUser -All $true
   Get-AzureADGroup -All $true

## APIs Used & When:

**Under the hood, calls Azure AD Graph or Microsoft Graph endpoints** (depending on module version).  
Used when an attacker has valid Azure AD credentials and the module is installed.

### Microsoft Graph API

Attackers with valid OAuth tokens can query user/group objects programmatically:

```bash
GET https://graph.microsoft.com/v1.0/users
GET https://graph.microsoft.com/v1.0/groups
```
Used if the attacker has **app registration** or **user tokens** with Graph permissions.  
Typically invoked from **custom scripts** or tools with an OAuth token.


## Soaphound (if it supports Azure AD):
```powershell
.\Soaphound.exe --enum-azuread --tenant yourtenant.onmicrosoft.com
```
**APIs Used & When**:

- **Possibly uses Microsoft Graph** or a custom SOAP-based endpoint.  
- **Used** when standard Azure modules are blocked or the attacker wants stealth via alternative protocols.
