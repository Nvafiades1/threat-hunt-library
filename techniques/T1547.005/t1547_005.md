## Overview

This detection identifies adversaries modifying **LSA-related registry values** that control which Dynamic Link Libraries (DLLs) are loaded by the Local Security Authority Subsystem Service (LSASS) on boot. The query looks for suspicious additions to the following multi-string (REG_MULTI_SZ) values:

* `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Security Packages`
* `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Authentication Packages`
* `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Notification Packages`
* `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Providers`

An attacker who inserts a malicious DLL name into any of these lists gains **SYSTEM-level persistence** and direct access to credential material once LSASS loads the DLL.

The logic relies on **CrowdStrike FDR** `AsepValueUpdate` events (ASEP = ÔÇťAuto-Start Extension PointÔÇŁ) which capture registry value changes. It parses the null-delimited string stored in `RegStringValue`, expands each DLL entry, and flags any DLL **not in a hardened baseline of legitimate Microsoft authentication packages**.

## Security Implications (ÔÇťwhy is this bad?ÔÇŁ)

1. **Persistent SYSTEM Code Execution** ÔÇô DLLs listed here load every boot inside LSASS.  
2. **Credential Theft** ÔÇô Code inside LSASS can read hashes, tickets, and plain-text passwords.  
3. **Privilege Escalation** ÔÇô LSASS runs as SYSTEM; attackers elevate automatically.  
4. **Defense Evasion** ÔÇô Living in a trusted process and abusing a legitimate mechanism.  
5. **Potential C2 / Exfil** ÔÇô Malicious SSPs can exfiltrate credentials or open network sockets from LSASS.  

## Detection Specifics

### Data Source

| Field | Purpose |
| :---- | :------ |
| `event_simpleName == "AsepValueUpdate"` | Registry value update to an Auto-Start location |
| `RegValueName` | Must equal one of the four LSA package value names |
| `RegStringValue` | Contains the **NULL-delimited** list of DLLs (parsed below) |
| `FileName` | Executable that made the change (for triage) |

### Baseline Used

```kusto
let Baseline = dynamic([
  "msv1_0.dll","kerberos.dll","schannel.dll","wdigest.dll",
  "tspkg.dll","pku2u.dll","credssp.dll","cloudap.dll","negexts.dll"
]);
```
### Detection Pattern (pseudo-logic)

1. **Filter** to `AsepValueUpdate` events where  
  `RegValueName` {`Security Packages`, `Authentication Packages`, `Notification Packages`, `Providers`}.
2. **Split** `RegStringValue` on `\u0000` (NULL). *Discard empty entries.*
3. **Extract** simple DLL names with  
  `tolower(split(dll_entry, "\\")[-1])`.
4. **Alert** when `DLLName` is **not** in **Baseline**.
5. **Provide** a human-readable preview string (`RegStringPreview`) by replacing NULLs with `; `.

#### Output Fields

- `TimeGenerated`
- `ComputerName`
- `FileName` (process that made the change)
- `RegValueName`
- `RegStringPreview` (user-friendly list)
- `RegStringValue` (raw, for full context)



## Existing Controls (expanded)

### 1. LSA Protection ÔÇô **RunAsPPL (Protected Process Light)**

| Aspect | Detail |
| :----- | :----- |
| **Purpose** | Prevents unsigned or lower-trust code from loading into LSASS. |
| **How It Works** | Setting `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\RunAsPPL = 1` causes **Smss.exe** to spawn LSASS as a *WinTcb-Light* protected process.<br>Kernel Mode Code Integrity (KMCI) validates every DLL before mapping; only Microsoft-signed or WHQL-signed binaries with the correct EKU are allowed.<br>If validation fails, the load returns **STATUS_INVALID_IMAGE_HASH** and an ETW 3065/3086 event. |
| **How to Verify It Is Enabled** | **Registry**: `RunAsPPL` (DWORD) = 1.<br>**BCDEdit**: `bcdedit /enum {current}` Ôćĺ look for `launchsecurityservices Yes`.<br>**Process Explorer**: LSASS shows **Protection: Protected**.<br>**PowerShell**: `(Get-Process lsass).ProtectionLevel` Ôćĺ **PsProtectedSignerWinTcb**. |
| **When It Works Correctly** | Secure Boot and HVCI are enabled; host has rebooted since registry change; OS Ôëą Windows 8.1. |
| **Limitations** | Malicious but trusted-signed DLLs still load.<br>Vulnerable drivers can disable KMCI before LSASS starts.<br>**Credential Guard does *not* block SSP loads**ÔÇöit only isolates secrets; a signed rogue SSP still executes inside LSASS. |



### 2. Application Control ÔÇô **WDAC / AppLocker / Third-Party Allow-listing**

| Aspect | Detail |
| :----- | :----- |
| **Goal** | Block unsigned or unapproved DLLs **before** they execute. |
| **Mechanism (WDAC)** | Deploy WDAC in **Audit Ôćĺ Enforce**.<br>Supplemental policy `LSA_SSP_Allow.xml` lists approved SSP hashes; enable **Options 13 & 16**. |
| **How to Verify It Is Blocking SSP Loads** | **Event Viewer** Ôćĺ *Applications and Services Logs Ôćĺ Microsoft Ôćĺ Windows Ôćĺ CodeIntegrity Ôćĺ Operational*:<br>ÔÇó **Event 3076**: DLL allowed.<br>ÔÇó **Event 3077/3078**: DLL blocked; shows rule ID.<br>**WDAC policy viewer** confirms policy ID matches. |
| **AppLocker (legacy)** | **Event 8004** in *Microsoft-Windows-AppLocker/Packaged-app-Deployment* signals DLL blocked by DLL rule. |
| **Third-Party** | EDRs expose their own telemetry (e.g., Carbon Black policy block event). |



### 3. Credential Guard ÔÇô **VBS Secret Isolation (does *not* block SSPs)**

| Aspect | Detail |
| :----- | :----- |
| **Purpose** | Moves hashes/TGTs out of LSASS into **LSAISO.exe** running in Virtual Secure Mode (VSM) to weaken post-exploitation credential theft. |
| **How It Works** | With VBS enabled and `LsaCfgFlags` = 0x1 or 0x2, LSASS brokers credential calls through a secure channel to the isolated process. |
| **Relationship to SSP Loads** | Credential Guard **does not** inspect or block DLLs entering LSASS.<br>A signed malicious SSP that clears RunAsPPL checks will still loadÔÇöeven with CG active.<br>The rogue SSP cannot read protected memory but can intercept plaintext creds via normal LSA callbacks. |
| **How to Block SSPs with CG (Win 11 24H2+)** | Set `LsaCfgFlags = 0x80000000`.<br>Windows refuses to load any third-party SSP/AP package (even Microsoft-signed but out-of-box).<br>ETW 3086 Ôćĺ **STATUS_INVALID_IMAGE_HASH**; DeviceGuard **Event 7001** logged. |
| **Limitations** | The 0x80000000 flag exists only on Windows 11 24H2+; earlier OSes rely on RunAsPPL + WDAC. |

## Known Gaps

1. **DLL Signed but Malicious** ÔÇô Requires reputation scoring or memory-scan follow-up.  
2. **Suppressed Telemetry** ÔÇô Host offline or sensor disabled; registry changes may be missed.  


## Long-Term Considerations

1. **Registry-to-Runtime Correlation** ÔÇô Alert when `AsepValueUpdate` is followed by a `ClassifiedModuleLoad` match within 30 min.  
2. **Hash-Pinned Baseline Service** ÔÇô Auto-refresh allow-list via Azure Function tied to Patch Tuesday.  
3. **YARA Memory Scans** ÔÇô Hourly LSASS scan for `SpInitialize`/`SpAcceptCredentials` in non-baseline modules.  
4. **Credential Guard Hard Enforcement** ÔÇô On Windows 11 24H2+, enable `LsaCfgFlags = 0x80000000` to prevent third-party SSP loads entirely.  
5. **Immutable Registry ACLs** ÔÇô Lock the four LSA values with a SACL allowing only TrustedInstaller.  
6. **Purple-Team Validation Cadence** ÔÇô Quarterly benign DLL drop; require < 15 min MTTD.  
7. **Outbound Behavior Guardrails** ÔÇô Companion analytics for LSASS opening sockets or spawning suspicious children.
