## Overview

This detection identifies adversary attempts to achieve persistence, escalate privileges, and access credentials by registering malicious Dynamic Link Libraries (DLLs) that are subsequently loaded by the Local Security Authority Subsystem Service (`lsass.exe`). This aligns with MITRE ATT&CK technique T1547.005.

The detection logic focuses on analyzing module load events (`ClassifiedModuleLoad` from CrowdStrike FDR) specifically for the `lsass.exe` process. It scrutinizes the characteristics of loaded DLLs, such as their signature status (unsigned, untrusted, or non-Microsoft) and their file path (e.g., loading from non-standard or high-risk locations), to identify potentially malicious LSA plugins like Security Support Providers (SSPs), Authentication Packages, or Notification Packages.

This DID is based on insights from **TRR0019: Persistence via Security Support Provider Registration**. While TRR0019 also details direct registry modifications and dynamic API calls (`AddSecurityPackage`), this detection primarily targets the observable artifact of `lsass.exe` loading these plugins.

## Security Implications ("why is this bad?")

Successfully registering and loading a malicious DLL into `lsass.exe` has severe security implications:

1.  **Persistent System-Level Access:** The malicious DLL is loaded automatically by `lsass.exe` on system boot (for registry-based methods), granting the adversary persistent code execution with `SYSTEM` privileges.
2.  **Credential Access:** Executing within the `lsass.exe` process space provides direct access to sensitive credential material, including NTLM hashes, Kerberos tickets, and potentially plaintext passwords (e.g., if WDigest is enabled or through API hooking). This facilitates further lateral movement and privilege escalation.
3.  **Privilege Escalation:** Code execution as `SYSTEM` is the highest level of privilege on a Windows host.
4.  **Defense Evasion:** Operating within the trusted `lsass.exe` process can help malicious code evade detection by some security tools. The use of legitimate LSA extensibility mechanisms is a form of legitimate functionality abuse.
5.  **Data Exfiltration & Command and Control:** The compromised `lsass.exe` process can be used to exfiltrate stolen credentials or establish covert C2 channels.

## Detection Specifics

This detection logic is designed for Anvilogic, utilizing Azure Data Explorer (ADX) with CrowdStrike FDR logs. It focuses on identifying suspicious or anomalous DLLs loaded by the `lsass.exe` process.

**Data Sources in ADX:**
* CrowdStrike FDR Events: `event_simpleName == "ClassifiedModuleLoad"` (Provides details on modules loaded by processes).

**Detection Pattern:**
The KQL query identifies `ClassifiedModuleLoad` events where the process loading the module (`TargetImageFileName`) is `c:\windows\system32\lsass.exe`. It then assesses the loaded DLL (`ImageFileName` for path, `FileName` for simple name) based on:

1.  **Signature Status (derived from `SignInfoFlags`):**
    * **Unsigned or Untrusted:** Flags DLLs that are explicitly unsigned, self-signed, have an invalid certificate chain, a hash mismatch, a failed certificate check, or an expired/revoked certificate. Modules with missing or unparsable signature information are also treated as untrusted.
    * **Non-Microsoft Signed:** Flags DLLs that are not verifiably signed by Microsoft, even if they have some other valid signature.
2.  **File Path:**
    * **High-Risk Paths:** Flags DLLs loaded by `lsass.exe` from user profiles, temporary directories, ProgramData, or other non-standard writable locations.
    * **Non-System32 Paths:** Flags DLLs loaded by `lsass.exe` from outside `C:\Windows\System32\`. While some legitimate third-party LSA plugins might reside in Program Files, these are less common and warrant scrutiny.
3.  **Suspicious Combinations:** An alert is generated if `lsass.exe` loads a module that is:
    * From a high-risk path, OR
    * Has an unsigned or untrusted signature, OR
    * Is not Microsoft-signed (as LSA plugins loaded by `lsass.exe` are overwhelmingly Microsoft components or need very specific justification if third-party).

While direct detection of registry modifications to `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\` keys (`Security Packages`, `Authentication Packages`, `Notification Packages`, `Providers`) is ideal for catching the persistence setup, this DID's KQL focuses on the module load due to prior difficulties in identifying specific registry event telemetry. Monitoring these registry keys through other means is highly recommended as a complementary detection. Similarly, direct API monitoring for `AddSecurityPackage` calls (Procedure E in TRR0019) would complement this module load detection.

## Known Benign Events
1.  **Legitimate Third-Party LSA Plugins:** Some security software (e.g., password filters, smart card drivers, custom authentication solutions) may install DLLs that are loaded by `lsass.exe`. These should ideally be digitally signed by the respective vendor. If they are consistently loaded from standard installation paths (e.g., `C:\Program Files\VendorName\`) and are signed by trusted vendors, they may need to be whitelisted by signer or specific path/hash after investigation. The current query will flag non-Microsoft signed DLLs.
2.  **Windows OS Updates/Patches:** Very rarely, an OS update might introduce or change a legitimate Microsoft-signed LSA module. These should be caught by the `IsMicrosoftSigned == true` condition and thus *not* be flagged by `(not IsMicrosoftSigned)`. If a legitimate MS DLL has problematic flags causing `IsMicrosoftSigned` to be false, that part of the logic may need tuning.
3.  **Diagnostic Tools:** Some advanced diagnostic or security assessment tools might (with sufficient privilege) query LSA packages or trigger loads, but these should be rare and authorized.

## Existing Controls
* **LSA Protection (RunAsPPL):** When enabled, this Windows feature restricts code injection into `lsass.exe` and requires LSA plugins (like SSPs) to be Microsoft-signed (or meet specific criteria for drivers). This is a strong preventative control.
* **User Account Control (UAC):** Helps prevent unauthorized privilege escalation needed to modify LSA registry keys or write to System32.
* **Endpoint Detection and Response (EDR):** Solutions like CrowdStrike Falcon provide visibility into module loads and process activity.
* **Registry Monitoring/Integrity:** Monitoring for unauthorized changes to the LSA package registry keys:
    * `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Security Packages`
    * `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Authentication Packages`
    * `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Notification Packages`
    * `HKLM\SYSTEM\CurrentControlSet\Control\Lsa\Providers`
* **File System Monitoring/Integrity:** Monitoring for suspicious DLLs being written to `%SystemRoot%\System32`.
* **Application Control:** Restricting which executables/DLLs can be loaded.

## Known Gaps
1.  **Sparse/Unreliable Signature Information:** If the `SignInfoFlags_Field` is frequently empty or cannot be reliably parsed for many legitimate or malicious modules, the signature-based aspects of this detection will be weakened. The query currently treats missing/unparsable signature info as "untrusted."
2.  **Direct LSASS Memory Manipulation:** This detection focuses on module loads. It will not detect adversaries who gain SYSTEM privileges and then directly manipulate LSASS memory to inject functionality or steal credentials without loading a distinct DLL via these mechanisms.
3.  **Sophisticated Evasion:** Extremely advanced attackers might find ways to bypass EDR module load telemetry or use techniques that don't trigger `ClassifiedModuleLoad` events as expected.
4.  **Non-Persistent `AddSecurityPackage` API Usage:** While this detection *might* catch the DLL loaded by `AddSecurityPackage` if it has suspicious path/signature characteristics, it doesn't detect the API call itself.
5.  **Registry Key Modification Not Directly Detected by this KQL:** This KQL focuses on the module load. Detecting the registry modification for persistence requires separate telemetry/rules (as mentioned in Detection Specifics).
6.  **Legitimate Non-Microsoft LSA Plugins:** The current suspicious logic (`not IsMicrosoftSigned`) will flag legitimate, signed, third-party LSA plugins. These require investigation and potentially whitelisting by specific characteristics (e.g., Signer if that data can be made available, or full path + hash).

## Long-Term Considerations
1.  **Improve Signature Data Reliability:** Investigate if more reliable/structured signer information (e.g., a dedicated `Signer` field) can be obtained from `ClassifiedModuleLoad` or by joining with other process events from CrowdStrike FDR. This would significantly enhance detection fidelity.
2.  **Baseline LSA Registry Keys:** Implement separate monitoring and alerting for any unauthorized or suspicious modifications to the LSA package registry keys detailed in TRR0019. Correlate these with `lsass.exe` module loads.
3.  **API Call Monitoring:** If possible, explore options for monitoring direct calls to `AddSecurityPackageW` via more advanced EDR capabilities or specific telemetry.
4.  **Whitelist Legitimate Third-Party LSA Plugins:** After investigation, develop specific whitelisting criteria for known good third-party LSA plugins based on their full path, SHA256 hash, and (ideally) digital signer.
5.  **User and Entity Behavior Analytics (UEBA):** Correlate suspicious `lsass.exe` module loads with other anomalous user or system behaviors.
