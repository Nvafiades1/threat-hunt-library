## Overview

This detection identifies attempts to access or copy critical Windows registry hives – specifically the Security Account Manager (SAM), SYSTEM, and SECURITY hives – from disk. Adversaries target these hives to extract local account password hashes (NTLM hashes) and other sensitive credential material. This activity is a key indicator of the MITRE ATT&CK technique T1003.002.

This Detection Implementation Document (DID) focuses on detecting the direct file access or copying of these hives, often a precursor to offline cracking or Pass-the-Hash attacks. The detection logic correlates file access events to these specific hives with suspicious process behavior, aiming for high-fidelity alerts.

This DID is supported by `TRR0001: OS Credential Dumping – Security Account Manager` (our previously discussed TRR).

## Security Implications ("why is this bad?")

Unauthorized access to the SAM and SYSTEM hives poses a significant security threat:

1. **Local Account Compromise:** Adversaries can obtain NTLM hashes for all local user accounts. These hashes can be:
   - **Cracked Offline:** Weak passwords can be quickly recovered, granting attackers plaintext credentials.
   - **Used in Pass-the-Hash (PtH) Attacks:** NTLM hashes can be used directly to authenticate to other systems and services within the network if credentials (username/hash) are reused, facilitating lateral movement.
2. **Privilege Escalation & Persistence:** Compromised local administrator accounts (especially the RID 500 account) provide high-level access and a durable method of persistence on the endpoint.
3. **Domain Compromise (Potentially):** If local administrator passwords/hashes are reused across multiple machines, including domain controllers (a poor practice, but it happens), this can escalate to domain compromise. The SECURITY hive may also contain cached domain credentials or LSA secrets.
4. **Bypassing Security Controls:** With valid credentials, attackers can operate more stealthily, appearing as legitimate users.

Successfully dumping these hives provides adversaries with valuable credential assets that significantly further their objectives within a compromised environment.

## Detection Specifics

This detection logic is designed to run in Anvilogic, utilizing ADX, and leveraging CrowdStrike FDR logs. It focuses on identifying suspicious read attempts (`FileOpenInfo`) of the SAM, SYSTEM, or SECURITY registry hive files.

**Data Sources in ADX:**
- CrowdStrike FDR Events: `event_simpleName == "FileOpenInfo"` (for file access)  
- CrowdStrike FDR Events: `event_simpleName == "ProcessRollup2"` (for process context)  
 

**Detection Pattern:**
1. **Non-Whitelisted Processes:** The process accessing the hive is not on a list of known legitimate system processes that normally interact with these files.  
2. **Suspicious Parent Processes:** The parent process is a common interpreter or scripting engine (`cmd.exe`, `powershell.exe`, `wscript.exe`, `explorer.exe`).  
3. **Process Masquerading:** Legitimate process names (e.g., `lsass.exe`, `svchost.exe`) running from non-standard paths (e.g., a temp directory).  
4. **Access to Backup Copies:** File extensions like `.bak`, `.sav`, or `.log` suggest use of backup/export copies.

A scoring mechanism combines these factors; alerts fire when `total_score >= 4`.

## KQL Detection Logic

```kql
// KQL for Anvilogic - T1003.002 SAM/SYSTEM/SECURITY Hive File Access

let HivePathRx = @"(?i)\\windows\\system32\\config\\(sam|security|system)(\.log|\.sav|\.bak)?$";
let SuspiciousParents = dynamic(["cmd.exe","powershell.exe","wscript.exe","explorer.exe"]);
let WhitelistedProcesses = dynamic([
  @"c:\windows\system32\lsass.exe",
  @"c:\windows\system32\services.exe",
  @"c:\windows\system32\winlogon.exe",
  @"c:\windows\system32\svchost.exe",
  @"c:\windows\system32\taskhostw.exe"
]);

let HiveAccess = materialize(
  database("Endpoint").CrowdStrikeFDR
  | where event_simpleName == "FileOpenInfo"
  | extend ActingProcessID = tolong(ContextProcessId),
           TargetFile      = tolower(tostring(TargetFileName))
  | where TargetFile matches regex HivePathRx
  | project TimeGenerated, ActingProcessID, TargetFile
);

let ProcContext = materialize(
  database("Endpoint").CrowdStrikeFDR
  | where event_simpleName == "ProcessRollup2"
  | extend pid_target   = tolong(TargetProcessId),
           proc_image   = tolower(ImageFileName),
           parent_image = tolower(ParentImageFileName),
           cmdline      = tostring(CommandLine)
  | project TimeGenerated, pid_target, proc_image, parent_image, cmdline
);

HiveAccess
| join kind=leftouter (
    ProcContext
  ) on $left.ActingProcessID == $right.pid_target
| where isnotempty(proc_image)
| extend isWhitelisted  = proc_image in (WhitelistedProcesses)
| extend bad_parent     = extract(@"[^\\]+$", 0, parent_image) in (SuspiciousParents)
| extend masquerade     = (
     (proc_image endswith "lsass.exe"   and not(proc_image startswith @"c:\windows\system32\")) or
     (proc_image endswith "winlogon.exe" and not(proc_image startswith @"c:\windows\system32\")) or
     (proc_image endswith "svchost.exe"  and not(proc_image startswith @"c:\windows\system32\")) or
     (proc_image endswith "services.exe" and not(proc_image startswith @"c:\windows\system32\")) or
     (proc_image endswith "taskhostw.exe" and not(proc_image startswith @"c:\windows\system32\"))
   )
| extend file_score  = case(
     TargetFile endswith ".bak" or TargetFile endswith ".sav" or TargetFile endswith ".log", 2, 1
   )
| extend proc_score  = iff(isWhitelisted, 0, 2)
| extend evade_score = iff(bad_parent or masquerade, 3, 0)
| extend total_score = 1 + file_score + proc_score + evade_score
| where total_score >= 4
| project
    DetectionTime         = TimeGenerated,
    TargetFile,
    SuspiciousProcess     = proc_image,
    ParentProcess         = parent_image,
    CommandLine           = cmdline,
    Score                 = total_score,
    IsWhitelistedProcess  = isWhitelisted,
    IsSuspiciousParent    = bad_parent,
    IsMasquerading        = masquerade
| sort by TimeGenerated desc
```
**Note:** (Note on KQL: The ContextProcessId from FileOpenInfo refers to the process that performed the action on the file. TargetProcessId in ProcessRollup2 typically refers to the process that is the subject of the ProcessRollup2 event itself. The join on ```$left.ActingProcessID == $right.pid_target``` assumes `TargetProcessId` from `ProcessRollup2` is the correct field for the process ID.)

## Known Benign Events
The following scenarios may trigger similar event patterns but are often legitimate. The whitelisting and scoring in the query aim to reduce these:

1.  **Legitimate Backup Software:** Some backup solutions might directly access these hive files or their Volume Shadow Copies. If these processes are known and consistent, they should be added to the `WhitelistedProcesses` list with their full paths, or more specific exclusion logic should be developed (e.g., based on command-line arguments or parent processes specific to the backup tool).
2.  **System Restore or Recovery Operations:** Legitimate system restore or recovery processes initiated by administrators might access these files. These are typically rare and might be correlated with authorized change tickets or interactive administrative sessions.
3.  **Windows Updates or Servicing:** Core Windows processes like `TrustedInstaller.exe`, `TiWorker.exe`, or `svchost.exe` (hosting specific services like Windows Modules Installer) may interact with registry hives during updates. The current whitelist includes `svchost.exe` from System32, and the added masquerade check for it helps improve fidelity. Correlate with patching cycles.
4.  **Security Scanners/EDR Agents:** Some legitimate security tools or the EDR agent itself might inspect these files as part of their operation. Ensure your EDR agent's own processes and other security tools are understood and appropriately whitelisted if they generate false positives.
5.  **Microsoft Distributed Transaction Coordinator (MSDTC):** `msdtc.exe` running from `C:\Windows\System32\` may legitimately access hive log files (e.g., `SYSTEM.LOG1`, `SECURITY.LOG1`). While the regex covers `.log`, the primary focus is on the main hives or full backups (`.bak`, `.sav`). If `msdtc.exe` activity on these log files becomes noisy, specific exclusions for it accessing only `.LOG` files might be needed, or ensure it's covered by whitelisting if it's always the System32 version.

For benign events, correlation with authorized change requests, planned maintenance windows, known legitimate software lists, and specific command-line arguments or parent processes is essential. The scoring model helps prioritize events where multiple suspicious indicators are present.

## Existing Controls

* **Endpoint Detection and Response (EDR):** CrowdStrike Falcon (or equivalent) provides visibility into process execution, file access, and other endpoint behaviors. It may have its own built-in detections for some variants of this technique.
* **Principle of Least Privilege (PoLP):** User accounts should operate with the minimum necessary privileges. Administrative privileges, required to access SAM/SYSTEM hives, should be tightly controlled.
* **Privileged Access Management (PAM):** Solutions to vault, manage, and monitor the use of privileged accounts.
* **Application Control/Whitelisting:** Restricting the execution of unauthorized executables can prevent custom tools or LOLBAS copied to writable locations from running.
* **Volume Shadow Copy (VSS) Access Controls & Monitoring:** Limiting who can create/manage VSS snapshots and monitoring for anomalous `vssadmin.exe`, `diskshadow.exe` usage or WMI calls related to VSS. This is often a precursor or companion to T1003.002.
* **Regular Credential Audits & Strong Password Policies:** Enforcing strong, unique passwords for local accounts makes offline cracking of dumped hashes significantly more difficult and time-consuming. Disabling LM hash storage is standard.
* **Host-Based Firewalls:** Can restrict remote access to services that might be exploited using compromised local credentials.
* **Security Awareness Training:** Educating users and administrators about the risks of credential reuse and phishing attacks that could lead to initial compromise.

## Known Gaps

1.  **Purely In-Memory Dumping (LSASS):** This detection focuses on file access to SAM/SYSTEM/SECURITY hives on disk. It will **not** directly detect attacks that dump credentials directly from LSASS memory (covered by T1003.001) or use other in-memory techniques to access hive data without typical file open operations.
2.  **Kernel-Mode Attacks / Raw Disk Access:** Advanced adversaries using kernel-mode rootkits or direct raw disk access to read hive files might bypass file system monitoring (`FileOpenInfo`). Detection for these requires different data sources or capabilities (e.g., Sysmon Event ID 9: RawAccessRead, or specialized EDR features).
3.  **Highly Evasive Custom Tools:** Sophisticated custom tools might employ advanced evasion techniques to either avoid generating the monitored file access events or to obfuscate their process lineage beyond the current checks.
4.  **Volume Shadow Copy (VSS) Extraction to Arbitrary Names:** While the query checks for common hive names and backup extensions, an attacker could copy a hive from a VSS snapshot to an arbitrarily named file that does not match `HivePathRx`. Detecting anomalous VSS creation/access itself is a crucial complementary detection.
5.  **Encrypted Hive Copies:** If an attacker encrypts the hive copies immediately after creation (e.g., before writing to a staged file), this detection would catch the initial access to the original hive but not necessarily the creation of the encrypted output if its name/path isn't suspicious.
6.  **Sophisticated `svchost.exe` or Other Whitelisted Process Abuse:** While masquerade checks for whitelisted processes running from non-standard paths are included, truly malicious code injected into a legitimate, correctly-pathed instance of `svchost.exe` (or other whitelisted processes) might still be missed by *this specific* path-based masquerade check. Deeper behavioral analysis (e.g., anomalous network connections, child processes, loaded modules from these processes) would be needed but significantly increases query complexity.
7.  **Delayed Process Context:** If the `ProcessRollup2` event providing context for the process that accessed the hive is significantly delayed beyond Anvilogic's join window or the implicit time correlation, the `leftouter join` might result in an event with missing process details, potentially impacting scoring or investigation.

## Long-Term Considerations

1.  **Enhance Host Context in Join:** If not handled automatically by Anvilogic, explicitly include a host identifier (e.g., `DeviceId`, `aid`) in the KQL join condition between `HiveAccess` and `ProcContext` to ensure accurate correlation in multi-host environments.
2.  **Integrate with VSS Monitoring Alerts:** Develop strong correlations between alerts from this detection and alerts for suspicious VSS creation, deletion, or modification activities (e.g., unusual `vssadmin.exe`, `diskshadow.exe` usage, or WMI VSS calls).
3.  **Develop/Refine LSASS Dumping Detections:** Ensure robust, separate detections are in place for T1003.001 (OS Credential Dumping: LSASS Memory). Consider correlating SAM/SYSTEM access with prior LSASS interaction.
4.  **Monitor for Raw Disk Access:** If available, integrate logs that show raw disk access (e.g., Sysmon Event ID 9) to detect attempts to read hives bypassing the file system.
5.  **UEBA Integration:** Feed alerts from this detection into a User and Entity Behavior Analytics (UEBA) system to correlate with other suspicious user or host activities and build a more comprehensive risk profile.
6.  **Continuous Whitelist & Parameter Tuning:** Regularly review and refine the `WhitelistedProcesses`, `SuspiciousParents` lists, and the `total_score` threshold based on observed activity, false positive rates, and known legitimate administrative tools/scripts in your evolving environment.
7.  **Automated Enrichment:** For generated alerts, automatically enrich them with host criticality, user roles, and recent related security alerts to aid SOC investigation.
8.  **SOAR Playbook Development:** Create or refine Security Orchestration, Automation, and Response (SOAR) playbooks to handle alerts from this detection, potentially including steps like endpoint isolation, further evidence collection, or ticket generation.
